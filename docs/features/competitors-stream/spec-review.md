## Вопросы / риски, которые остаются открытыми

1. **`UPDATE` без WHERE на production** — намеренно по решению 1a. Безопасно при условии что миграция применяется до первого нового `INSERT INTO research_jobs`. Если параллельно с миграцией в production летят новые intake — есть теоретический race. Митигация: применяем миграцию в момент низкого трафика (сейчас приложение в разработке, реальных клиентов нет).

2. **PostgreSQL `ALTER TYPE ... ADD VALUE`** не может быть в той же транзакции что и использование значения. Drizzle обычно справляется, но если миграция упадёт на этом — придётся разбить на 2 SQL-файла.

3. **Strategy-промпт не меняем в этой задаче** — он уже умеет выдавать карточки конкурентов из предыдущего коммита. После добавления `competitorsBlock` в RAG он автоматически получит более релевантные данные.
