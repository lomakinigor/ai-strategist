# Brainstorm: Two-stage strategy generation

Дата: 2026-05-01

## Проблема

Текущая генерация стратегии — один LLM-вызов с max_tokens=5000, упирается в Vercel 60s timeout (Hobby план). Качество страдает из-за вынужденного сокращения; 504 случается на длинных контекстах.

## Цель

Разбить генерацию на этапы с параллелизацией и промежуточной валидацией качества каждой секции перед синтезом.

## Discovery questions

### 1. Кто пользователь и что он делает?

Владелец проекта (на стадии разработки). Хочет проверять качество каждой секции (бизнес/рынок/аудитория/каналы/конкуренты) **до** того, как они уйдут в синтез общей стратегии. Если секция плохая — увидит это сразу, не дожидаясь финального отчёта.

### 2. Какое поведение должно измениться?

- Кнопка «Сгенерировать стратегию» запускает Stage 1: 5 параллельных LLM-вызовов
- После Stage 1 пользователь видит 5 секций на странице отчёта + кнопку «Запустить синтез общей стратегии»
- Кнопка синтеза → Stage 2: 6-й LLM-вызов, который получает 5 готовых секций + RAG-context и пишет финальный раздел «Стратегия и рекомендации»
- После синтеза artifact в финальном статусе `done`, contentMarkdown содержит весь отчёт

### 3. Что НЕ должно измениться?

- Структура промптов (анти-платитюды, asset-card pattern) — каждый раздел продолжает использовать те же правила
- API контракт `generateStrategyAction(formData)` — тот же server action
- Mock-режим (без OPENROUTER_API_KEY) — единый mock-результат, без двух стадий
- Существующие unit-тесты `parseSections`, `getMockDraft` — не падают

### 4. Какие обходные пути это закрывает?

- Pro Vercel ($20/мес) → не нужен сейчас
- Inngest/фоновые задачи → не нужны для MVP
- Сокращение max_tokens до 3500 (что обеднит отчёт) → не нужно

### 5. Какие риски / unknown?

- **Параллельность**: 5 параллельных вызовов в один момент → провайдеры OpenRouter могут throttle. Митигация: `provider.allow_fallbacks: true` уже включено
- **Контекст для синтеза**: 5 готовых секций (~2000-3000 токенов каждая) + RAG-блоки = большой ввод. Может тормозить синтез
- **Ошибка одной из 5 секций**: блокирует Stage 2 (по решению Q3 в спеке)
- **Артефакт между стадиями**: пользователь может закрыть страницу — нужно корректно восстановить состояние при возврате

### 6. Как проверить успех?

- Stage 1 успевает за ≤45s (с запасом до 60s)
- Stage 2 успевает за ≤45s
- При `STRATEGY_TWO_STAGE_REVIEW=true` UI показывает 5 секций и кнопку синтеза
- При `STRATEGY_TWO_STAGE_REVIEW=false` (или не задано) — fallback на старое поведение (один вызов)
- Возврат на страницу `/research/[id]/report?artifactId=X` с `status='partial'` показывает 5 секций и кнопку синтеза
- При ошибке одной из 5 секций — кнопка синтеза заблокирована, видна кнопка ретрая упавшей секции

## Зафиксированные ответы пользователя

- Q1: B (5 параллельных + синтез после) — подтверждено
- Q2: B (один статус, без поллинга) — подтверждено в общей форме, но добавлен этап `partial` для двух стадий
- Q3: A с retry — блокируем синтез если упала секция, retry per-section
- Stage показ: ВКЛЮЧЕН на стадии разработки, выключаемо через env флаг
