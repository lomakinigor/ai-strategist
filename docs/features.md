# Features — ai-strategist

> Фичи описывают крупные блоки функциональности. Каждая фича разбивается на задачи в [tasks.md](tasks.md).

## F-000 — Project Foundation

**Статус:** Done  
**Описание:** Инициализация проектной базы — документация, правила, структура, память проекта.  
**Что включает:** CLAUDE.md, .claude/rules/, MEMORY.md, docs/, knowledge/  
**Что НЕ включает:** бизнес-логику, UI, API, базу данных

## F-001 — Data Model & Infrastructure

**Статус:** Planned  
**Описание:** Схема базы данных и настройка проекта. Сущности: компания, источник, факт, задание на исследование, отчёт.  
**Технологии:** Drizzle ORM, Postgres, pgvector  
**Результат:** миграции, TypeScript-типы, базовые CRUD-операции

## F-002 — Reliability Engine

**Статус:** Planned  
**Описание:** Модуль верификации данных. Присваивает RS (1–5), классифицирует утверждения: ФАКТ / ГИПОТЕЗА / НЕДОСТАТОЧНО ДАННЫХ. Определяет уровень достоверности HIGH / MEDIUM / LOW.  
**Результат:** функции классификации, типы, тесты

---

## F-003 — Minimal Research Intake

**Статус:** Planned  
**Описание:** Минимальная форма загрузки информации о компании. Клиент заполняет ~5 полей: название, отрасль, описание, сайт, каналы присутствия. После отправки — автоматически запускается research pipeline.  
**UX-принцип:** минимум вопросов. Никаких дополнительных уточнений в процессе, если не критично.  
**Результат:** UI-форма, валидация, сохранение в БД, запуск pipeline

---

## F-004 — Auto-Research Pipeline

**Статус:** Planned  
**Описание:** Автоматический сбор данных по 4 направлениям после intake. MVP — mock-адаптеры. Реальные провайдеры добавляются позже.

**4 потока исследования:**
1. **Business research** — компания: продукты, позиционирование, конкуренты, вакансии
2. **Market research** — рынок и ниша: объём, тренды, динамика (РФ + мировые с пометкой)
3. **Audience research** — целевая аудитория: сегменты, поведение, потребности, боли
4. **Channels research** — каналы присутствия: активность, охват, сравнение с конкурентами

**Интерфейс адаптера:** `{ data, source, date, rs, research_type }`  
**Результат:** 4 mock-адаптера, единый интерфейс, структура готова для реальных провайдеров

---

## F-005 — Validation Workspace

**Статус:** Planned  
**Описание:** UI для просмотра собранных данных перед генерацией отчёта. Факты сгруппированы по 4 категориям (бизнес / рынок / аудитория / каналы). Клиент может отметить факты как неактуальные.  
**Результат:** страница с фильтрацией по категориям, RS, типу факта + источниками

---

## F-006 — RAG Context Layer

**Статус:** Planned  
**Описание:** Векторное хранилище фактов по компании. Embeddings → pgvector. Контекст формируется из собранных данных, не из внутренних знаний LLM.  
**Результат:** pipeline: факты → embeddings → pgvector → retrieval по research_type

---

## F-007 — AI Strategy Generation

**Статус:** Planned  
**Описание:** Генерация структурированного отчёта через Vercel AI SDK. Входные данные — верифицированные факты из RAG по всем 4 потокам. LLM не добавляет неподтверждённые утверждения. Fallback-уточнение клиента — только при критической нехватке данных.  
**Результат:** streaming-генерация, структурированный вывод по 5 разделам

---

## F-008 — Strategy Workspace

**Статус:** Planned  
**Описание:** UI для просмотра готового отчёта.  
**Разделы отчёта:**
1. Исследование бизнеса
2. Исследование рынка
3. Исследование аудитории
4. Анализ каналов
5. Стратегия и рекомендации (с уклоном в автоматизацию и AI)

Маркировка ФАКТ / ГИПОТЕЗА / НЕДОСТАТОЧНО ДАННЫХ для ключевых утверждений.  
**Результат:** читаемый, структурированный отчёт на русском

> ⚠️ **Формат вывода — рабочий и временный.** MVP использует нейтральный, практичный, структурированный формат. Детальная проработка структуры блоков, стандартов клиентской подачи и уровня детализации — отдельный post-MVP этап после первых тестов (см. F-010).

---

## F-009 — Export Foundation

**Статус:** Planned  
**Описание:** Базовая инфраструктура для будущего экспорта (PDF, DOCX, Markdown). MVP — только заглушка и структура данных. Модуль изолирован от ядра — его можно переработать без затрагивания research pipeline или RAG.  
**Результат:** интерфейс экспорта, mock-реализация

---

## F-010 — Report Output Refinement

**Статус:** Future (post-MVP)  
**Описание:** Глубокая доработка клиентского отчёта после появления первых реальных тестов. Включает: структуру разделов, логику блоков, уровень детализации, стандарты клиентской подачи, форму рекомендаций.

**Запускается только после:** хотя бы одного полного цикла с реальным клиентом или реальными данными.

**Что НЕ входит в MVP:** финальные правила формата, визуальное оформление, шаблоны блоков.  
**Результат:** зафиксированные правила report output, обновлённые промпты и UI компоненты
