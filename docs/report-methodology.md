# Методология полного отчёта — decision-driven форма §0–8

Утверждена 2026-05-26 по итогам итераций GLC «Репутация» (v0.1 → v0.2). Это форма **Этапа 1** двухэтапной методологии (FULL_REPORT). Этап 2 (BRIEF_REPORT) — дистилляция из неё, см. [CLAUDE.md](../CLAUDE.md).

## Зачем форма меняли

Старый отчёт был **тематически-описательным** (8 разделов: профиль / рынок / ЦА / каналы / конкуренты + синтез). Из-за этого:

- один и тот же факт переописывался в нескольких разделах (повторы);
- много очевидных фактов без вытекающего действия;
- AI-рекомендации разбросаны и не убедительны;
- каналы перечислены, но не ранжированы под нишу;
- из отчёта не следовало, **что делать** для достижения целей.

Новая форма — **решенческая**: организована не по темам, а по решениям. Факт существует в отчёте только если из него следует действие.

## 7 принципов формы

1. **Пирамида Минто.** Каждый раздел начинается с вывода/решения, не с описания.
2. **Факт — один раз.** Все ключевые факты раскрываются только в §1 «Диагноз». Дальше по тексту — ссылка «см. §1», без переописания. Повтор факта между разделами = ошибка.
3. **Факт без действия не включается.** Очевидная фактура, из которой ничего не следует, выкидывается.
4. **Ранжирование.** Ходы, каналы, AI-рычаги — упорядочены по приоритету (impact/effort), а не списком.
5. **Привязка к цели.** Каждое направление связано с целью клиента.
6. **AI — следствие узких мест.** Каждый AI-рычаг ссылается на конкретное узкое место из §1. Никакого «AI ради AI».
7. **Ранжированный микс под нишу + программа тестов.** Обязательны: маркетинговый микс, заточенный под отрасль, и набор экспериментов с двумя исходами (масштабировать / остановить).

## Структура §0–8

| § | Раздел | Что внутри | Носитель | Методология |
| --- | --- | --- | --- | --- |
| 0 | Резюме для собственника | тезис (где сейчас + главное ограничение); 3 уровня действий иерархией (Фундамент / Быстрый старт / Умножитель); цена бездействия | абзац + таблица | Пирамида Минто |
| 1 | Диагноз роста — узкие места | 3–5 УМ: факт [с маркировкой] → последствие → «Ведёт к §X». Единственное место, где факты раскрыты подробно | текст | **Porter's 5 Forces** + **PESTEL** |
| 2 | Позиционирование и аудитория | решение (какую нишу занять); 2–3 сегмента; отстройка по типам конкурентов; сообщение первого экрана | текст + таблица сегментов | **JTBD** + **Blue Ocean Value Curve** + **Ries/Trout** |
| 3 | Маркетинговый микс под нишу | ранжированная таблица каналов (CPL-ориентир, бюджет/мес, что тестируем) + итог бюджета + «чего НЕ делаем» | таблица | 4P/STP |
| 4 | AI-автоматизация | таблица: узкое место → рычаг → механика → эффект → стоимость → срок; строка ROI. Каждый рычаг привязан к УМ | таблица | ROI-обоснование |
| 5 | План действий по горизонтам | неделя (0 ₽) / 30–60 дней / 3–6 месяцев; каждое действие → УМ → owner → метрика | таблицы по горизонтам | **McKinsey 3 Horizons** |
| 6 | Программа тестов продвижения | T1…Tn: гипотеза / как тестируем / метрика / срок / при успехе / **при провале** | таблица | Lean Startup Build-Measure-Learn |
| 7 | Риски и меры | риск / сигнал / мера (3–4 строки) | таблица | Risk Heat Map |
| 7.5 | Что не покрывает этот отчёт | честный disclaimer о границах формата (нет CustDev, нет доступа к CRM, снимок во времени, только публичные данные, не сопровождаем внедрение) | список | прозрачность ожиданий |
| 8 | Источники | факт / источник / дата / RS — для опорных [ФАКТ] из §1 и §3 | таблица | Reliability Score 1–5 |

**Зачем именованные методологии в §1, §2, §5:** клиент пересылает отчёт своему маркетологу/консультанту. Знакомые имена методологий (Porter, JTBD, McKinsey 3H) показывают, что отчёт построен на узнаваемой академической базе, а не на «мнении ИИ». Это снимает претензии «откуда такие выводы».

**Зачем §7.5 «Что не покрывает»:** честные границы формата снимают завышенные ожидания и одновременно повышают доверие — авторы отчёта думали о пределах применимости, а не выдают за «комплексный анализ всего». Без cliffhanger на L3 — просто честность.

**Объём:** 1500–2500 слов. Документ для принятия решений — плотно, без воды. Длинный отчёт с повторами = провал задачи.

**Носитель:** таблицы + плотный текст. НЕ «карточки активов» (🏢 ✅ ❌ 🎯) — это формат старой формы.

## Дисциплина достоверности (поверх формы)

- Каждое утверждение помечено: **[ФАКТ]** (RS ≥ 3) · **[ГИПОТЕЗА]** · **[НЕДОСТАТОЧНО ДАННЫХ]**. Подробнее: [.claude/rules/data-reliability.md](../.claude/rules/data-reliability.md).
- §8 «Источники» фиксирует у каждого опорного факта `источник · дата · RS`. Где неизвестно — «требует подтверждения», без выдуманных значений.
- Мировые бенчмарки — с пометкой «применимость к РФ не гарантирована».
- Все суммы — только в ₽. Текст — на русском, термины со скобочной расшифровкой при первом упоминании.
- Числовые замеры (PageSpeed/Lighthouse) брать из `research_jobs.metrics_json`, чтобы отчёт и дашборд не расходились. Lab-данные PageSpeed гуляют ±3–5 между прогонами — привязывать к одному замеру с датой.

## Метод применения к новому клиенту (7 шагов)

Абстракция формы, переносимая на любую нишу:

1. **Сбор и бенчмаркинг** — бизнес-цели; цифровые метрики (PageSpeed, CPL, конверсия); отраслевые нормы.
2. **Разрыв** — «где сейчас» vs «где должны быть» по ключевым KPI.
3. **Диагноз** — 3–5 узких мест (воронка, позиционирование, инфраструктура), каждое с фактом-основанием.
4. **Фокус** — какую нишу/сегмент занять; отстройка от типов конкурентов.
5. **Микс + AI** — ранжированные каналы под отрасль; AI-рычаги на узкие места.
6. **Тесты** — 4+ эксперимента с метрикой, сроком и двумя исходами.
7. **Управление** — план по горизонтам с owner/метрикой, риски с мерами, источники.

## Где это в коде

- Промпт формы: [src/lib/strategy/prompts.ts](../src/lib/strategy/prompts.ts) — `FULL_REPORT_SECTION_STRUCTURE` + `buildFullReportPrompt`.
- Генерация: [src/lib/strategy/generator.ts](../src/lib/strategy/generator.ts) — `generateStrategyDraft` (ветка one-shot), `classifySectionTitle` (маппинг заголовков → `SectionId`, новая форма + legacy).
- Требования базы знаний (подставляются в промпт): [knowledge/report-requirements/](../knowledge/report-requirements/) — `universal.md` + файл ниши.
- Рендер: [app/research/[id]/report/page.tsx](../app/research/%5Bid%5D/report/page.tsx) — лейблы/цвета по `SectionId`.

**Режим генерации:** один проход (single-pass). Требует `STRATEGY_TWO_STAGE_REVIEW=false` (иначе включится legacy two-stage путь со старой тематической формой).

**Провайдер:** OpenRouter → Claude Sonnet 4.6 (`synthesisModel`), один вызов, лимит `FULL_REPORT_MAX_TOKENS`.

## Обратная совместимость

Старые отчёты (тематическая форма §1–8) продолжают рендериться: `classifySectionTitle` распознаёт и старые, и новые заголовки. Новые отчёты генерируются только в форме §0–8.
