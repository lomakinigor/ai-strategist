# Универсальные требования — все ниши

> Полный отчёт строится в **decision-driven форме §0–8** (решенческая, не тематическая).
> Методология: [docs/report-methodology.md](../../docs/report-methodology.md).

## Обязательные разделы (форма §0–8, любая ниша)
Отчёт организован по решениям, не по темам. Все разделы присутствуют без исключений:
- **§0 Резюме для собственника** — тезис + 3 уровня действий иерархией (Фундамент / Быстрый старт / Умножитель) + цена бездействия.
- **§1 Диагноз — узкие места** — 3–5 УМ, каждое: факт → последствие → куда ведёт. Факты раскрываются ТОЛЬКО здесь (один раз).
- **§2 Позиционирование и аудитория** — свободная ниша; минимум 2 сегмента с триггерами; **конкуренты названы конкретно** (если клиент не указал — найти и показать, пометка `[найден AI]`); сообщение первого экрана. Несколько направлений бизнеса — позиционировать раздельно. **Конкуренты — в 6-уровневой схеме fan-out** (см. ниже отдельный раздел).
- **§3 Маркетинговый микс под нишу** — **оценить ВСЕ перспективные для ниши каналы**, каждый с вердиктом «рекомендуем / не сейчас + почему» (молча опускать нельзя). Ранжированная таблица: CPL-ориентир, бюджет/мес в ₽, что тестируем + итог бюджета + «чего НЕ делаем». Авито / маркетплейсы / 2ГИС оценивать по нише, не занижать. Разделять «используется клиентом» (только при подтверждении) и «рекомендуется».
- **§4 AI-автоматизация** — таблица «узкое место → рычаг → механика → эффект → стоимость → срок» + строка ROI. Каждый рычаг привязан к УМ из §1.
- **§5 План действий** — 3 горизонта (неделя 0 ₽ / 30–60 дней / 3–6 месяцев); каждое действие → УМ → owner → метрика.
- **§6 Программа тестов** — минимум 4 теста: гипотеза / метрика / срок / **при успехе / при провале** (два исхода).
- **§7 Риски и меры** — таблица «риск / сигнал / мера» (3–4 строки).
- **§8 Источники** — таблица «факт / источник / дата / RS» для опорных фактов из §1 и §3. **Anti-vibe-citing protocol — обязательно** (см. отдельный раздел ниже).

## Ключевые принципы (нарушение = отчёт «как раньше»)
- Факт заявляется ОДИН раз (в §1), дальше — ссылка «см. §1». Никаких повторов между разделами.
- Факт без вытекающего действия в отчёт не включается.
- Каналы и AI-рычаги ранжированы и привязаны к узким местам.
- Объём: 1500–2500 слов, плотно, без воды. Носитель — таблицы, не «карточки».

## Обязательная AI-рекомендация для ВСЕХ компаний (полный И краткий отчёт)
Эта рекомендация всегда входит в блок AI-автоматизации, независимо от ниши:
- **Автоматизация контента**: AI-генерация постов и новостей сайта + автопостинг по каналам,
  которые указал сам клиент (соцсети, сайт, email, мессенджеры). Каналы РФ в приоритете:
  MAX → ВКонтакте → Telegram → email.
- Эффект указывать в цифрах: экономия часов/мес и/или ₽/мес (в т.ч. вместо найма SMM-специалиста,
  ЗП в РФ 60 000–120 000 ₽/мес).
- Если у клиента уже есть автоматизация контента [ФАКТ] — оценить качество и предложить доработки.
- Эта рекомендация ОБЯЗАНА попасть и в полный отчёт (раздел AI-автоматизация), и в краткий
  (блок AI-рычаги). Если её нет — отчёт неполный.

## 6-уровневая схема конкурентов в §2 (любая ниша)

**Fan-out:** research-пайплайн запускает отдельный вызов на КАЖДОГО названного клиентом
конкурента (кап 6 — защита от расходов). Каждый вызов идёт с чистым контекстом и единой
схемой — это устраняет дрейф качества, который был у single-agent при разборе 4+
конкурентов подряд. Реализация: `parseCompetitorNames(query.competitors)` в
`src/lib/research/competitor-names.ts` + параллельные вызовы `researchType='competitor_single'`
в orchestrator. Факты возвращаются как `researchType='competitors'` (тот же bucket контекста).

**6 уровней (каждый конкурент):**

- **Offer** — конкретная услуга / формат / обещание (с цитатой и URL).
- **Audience** — на какой сегмент B2B/B2C нацелен (МСБ / корпорации / ИП / физлица).
- **Pain** — какую боль адресует.
- **Proof** — соц.доказательства: цифры, клиенты, отзывы, рейтинги, награды, лицензии.
- **Creative** — визуальный стиль лендинга и тон коммуникации.
- **Landing structure** — структура первой страницы.

**Синтез §2 при N подтверждённых конкурентах:**

1. Сводная таблица N×6 (столбцы — конкуренты, строки — 6 уровней).
2. **Паттерны** — что делают 3+ одинаково (это базовая гигиена, НЕ отстройка).
3. **Белые пятна** — что НЕ делает никто (это и есть свободное место).
4. **2–3 testable-гипотезы** в формате «Если X → Y, то Z, потому что никто из N не делает W» —
   нумерация сквозная (H1, H2, H3), переходят в §6 «Программа тестов».
5. Короткая таблица позиционной отстройки | Конкурент | Метка | Где сильнее | Наша отстройка |.

Если N=0 (клиент не указал) — старая схема: одна таблица «Конкурент / Что делает / Наша отстройка»,
конкурентов берёт AI из общего `competitors`-потока с пометкой [найден AI].

## Обязательная проверка llms.txt (полный И краткий отчёт, любая ниша)
**Безусловное правило:** для каждой компании проверяется наличие файла `llms.txt` на её сайте
(стандарт 2024 для AI-агентов — структурированное описание сайта для LLM, помогает корректному
цитированию ChatGPT/Perplexity/Claude). Проверка реализована в `src/lib/research/site-crawl/`
и эмитит факт [ФАКТ][RS:4] о наличии/отсутствии независимо от других результатов краула.
Статус ОБЯЗАН быть отражён:
- **Полный отчёт (§4 AI-автоматизация):** если отсутствует — отметить как разрыв AI-readiness
  и включить «опубликовать llms.txt» в §5 «План действий» на горизонт «Неделя» (быстрая,
  дешёвая мера). Если присутствует — отметить как положительный сигнал AI-readiness.
- **Краткий отчёт:** если отсутствует — «Опубликовать llms.txt» в `next_actions` (короткий
  дедлайн, KPI «опубликован»). Если присутствует — упомянуть в `market_position` или
  `ai_levers` как AI-readiness сигнал.
- **§8 Источники полного отчёта:** URL вида `https://домен/llms.txt` обязателен.

## Обязательная проверка AI-агентов / чат-ботов (полный И краткий отчёт, любая ниша)
**Безусловное правило:** для каждой компании любой ниши пайплайн проверяет присутствие
чат-виджетов / AI-агентов / квалификационных ботов на сайте клиента (и в каналах, если
клиент их указал). Реализовано в `src/lib/research/site-crawl/ai-agents.ts` (детектор
по сигнатурам топовых РФ + международных платформ: Jivo, Bitrix24, RedHelper, Talk-me,
Envybox, Callibri, Marquiz, AmoCRM, OpenAI и др.) → факт [ФАКТ][RS:4] с источником-URL
главной. Статус ОБЯЗАН быть отражён:
- **Полный отчёт (§4 AI-автоматизация):** если отсутствует — пункт «Разработать и
  встроить AI-агента для квалификации лидов во ВСЕ каналы клиента (сайт + перечисленные
  клиентом)»; эффект — снижение CPL, разгрузка менеджеров, ответы 24/7; §5 «План
  действий» — горизонт 30–60 дней. Если найден чат-виджет — уточнить, операторский он
  или AI-powered; операторский ≠ AI-агент → рекомендация апгрейда. Если AI-powered
  подтверждён — отметить как сильный сигнал.
- **Краткий отчёт:** отсутствие → один из 3 `ai_levers` = «AI-агент для квалификации
  лидов». Операторский чат → «Апгрейд чата до AI-агента». AI-powered → упоминание
  в `market_position` как сильный сигнал.

## §8 Источники — Anti-vibe-citing protocol (Level 1, обязательно)

> **Контекст:** в июне 2026 GPTZero опубликовал исследование, в котором проверил 45 ссылок в собственном отчёте KPMG об ИИ-агентах — только 5 оказались валидными, **40 (89%) — поддельные или сильно искажённые**. Феномен назван **vibe citing** (по аналогии с vibe coding). Прецедент 2025: вымышленные сноски в докладе MAHA правительства США. AI-Стратег применяет анти-vibe-citing protocol уже на промпт-уровне (Level 1) и автоматическую проверку ссылок в Level 2-3 (на roadmap).

При формировании §8 «Источники» обязательно соблюдать 5 правил:

1. **Точные URL из web_search.** Для каждого факта в §1-§7 указывай URL, дата доступа (формат YYYY-MM-DD), точная цитата 1-2 строки. URL копируй ИЗ результата web_search, не из памяти модели и не реконструируй из доменного имени.
2. **Не выдумывать.** Запрещено добавлять DOI, ISBN, точный год публикации без верифицированного источника. Если не уверен — указывай только URL + дата доступа. Лучше короткая ссылка, чем выдуманные метаданные.
3. **Имена авторов — как в исходнике.** Не «улучшать» написание, не править предполагаемые опечатки. AI-системы часто перепутывают близкие имена или меняют авторов местами — это первый признак vibe citing в чужих отчётах.
4. **Не «улучшать» названия работ.** Цитируй точное название публикации. Перефразирование названий — второй признак vibe citing.
5. **При сомнениях — source-confidence: LOW.** Если в любом параметре источника (URL / автор / название / год / выпуск) есть неопределённость — рядом с источником добавляй пометку `source-confidence: LOW`. Лучше явно отмечен low-confidence источник, чем уверенно ложный.

**Признаки vibe citing в чужих отчётах** (если AI-Стратег анализирует чужой материал, например конкурента):

- Одна и та же ошибка повторяется в разных разделах (имя автора перепутано везде одинаково)
- Названия работ «слегка не такие», как у реальных публикаций по теме
- DOI/URL/года не сходятся при попытке проверить хоть одну ссылку
- Большой процент 404 при выборочной проверке
- «Удобные» цифры с апелляцией к авторитетам без точного года и conference proceedings

Если 2+ признака подтверждены — относиться к чужому материалу как **«vibe-cited»**, требовать оригинал или дисквалифицировать утверждения как недоказанные.

**Маркировка в финальном отчёте AI-Стратега:** под таблицей §8 рендеринг добавляет автоматический callout «✅ Все N источников проверены автоматически перед публикацией отчёта (дата). Anti-vibe-citing protocol: каждый URL extracted из web_search». Это формирует expectation у клиента и позиционирует продукт.

**Триггер ошибки (для будущего автоматического Red-Team шага Level 2):** жёсткий порог >5% мёртвых ссылок = отчёт не выпускается клиенту. Сейчас (Level 1) — превентивная защита промптом, реальный fetch будет в Level 2.

## Маркировка данных (обязательно для каждого факта)
- ФАКТ — есть источник, цифра подтверждена, reliability HIGH/MEDIUM
- ГИПОТЕЗА — логичное предположение, требует проверки
- НЕТ ДАННЫХ — информации недостаточно, нужен сбор

## Правила компоновки (Minto Pyramid)
- Главный вывод — первым предложением каждого раздела
- Доказательства и детали — после вывода
- Никогда не начинать раздел с предпосылок или истории

## Правила визуализации данных (Knaflic — Storytelling with Data)
- Таблицы только с контекстом: было → стало
- Метрики рядом с нормой отрасли
- Статусы: 🔴 критично / 🟡 требует внимания / 🟢 норма

## Принцип "и что мне с этого?" (Duarte — Resonate)
Каждый раздел заканчивается:
- Конкретным действием (глагол + объект)
- С дедлайном (конкретная дата или срок)
- С измеримым результатом (цифра или метрика)

## Правила краткого отчёта (BRIEF_REPORT)
- Только ФАКТ — гипотезы без подтверждения убираются
- Каждое утверждение содержит цифру
- Тест: читается за 3 минуты? Если нет — сократить
- Объём: строго 600–900 слов
