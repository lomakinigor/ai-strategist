# ai-strategist — CLAUDE.md

## Идентичность проекта

**ai-strategist** — SaaS-платформа для российских компаний. Клиент заполняет минимальную форму о своём бизнесе, после чего система автоматически запускает исследовательский pipeline и формирует:
1. Исследование бизнеса клиента
2. Исследование рынка и ниши
3. Исследование целевой аудитории
4. Анализ каналов и присутствия
5. Стратегию развития компании с рекомендациями по автоматизации и AI

## Ключевой принцип

> Каждый факт — с источником, датой и reliability score. Нет данных — нет вывода.

Подробнее: [.claude/rules/data-reliability.md](.claude/rules/data-reliability.md)

## Продуктовый скоуп MVP

**Входные данные:** минимальная форма о компании (intake)  
**Процесс:** автоматический research pipeline — без лишних вопросов клиенту  
**Уточняющие вопросы:** только fallback, если критически не хватает данных для анализа  
**Выходные данные:** аналитический отчёт + стратегические рекомендации (PRD → [docs/PRD.md](docs/PRD.md))  
**Формат отчёта:** нейтральный, практичный, структурированный — будет уточняться после запуска MVP  
**Интеграции MVP:** mock/in-memory first, потом реальные провайдеры

## Предпочтительный UX-принцип

> Минимальное количество вопросов клиенту и автоматический сбор максимально возможного объёма данных после intake.

MVP — это **автоматизированный стратегический исследовательский workflow**, а не чат-интервью. После intake система сама проводит все 4 направления research и формирует отчёт.

## Принцип клиентского отчёта

> **Клиентский отчёт — главный результат работы приложения.**

Вся техническая цепочка (intake → research → RAG → LLM) существует ради одного: чтобы клиент получил качественный, достоверный, практически полезный отчёт.

**MVP-формат отчёта — рабочий и временный.** Финальные правила структуры, логики блоков, уровня детализации и стандартов клиентской подачи **не фиксируются до появления первых реальных клиентских отчётов**. После первых тестов — отдельный этап глубокой доработки output format.

## Формат ответа

```
1. Brief analysis
2. Implementation plan
3. Files created/changed
4. Implementation summary
5. Recommended next step
6. РЕЗЮМЕ ДЛЯ ПЕРЕДАЧИ В ЧАТ
```

## Обязательное резюме (жёсткое правило)

> **Задача считается завершённой только если ответ содержит блок «РЕЗЮМЕ ДЛЯ ПЕРЕДАЧИ В ЧАТ».**

Claude обязан сгенерировать этот блок до остановки. Если основная работа выполнена, но блока нет — задача не завершена.

**Минимальный состав резюме:**
- Какие файлы созданы/изменены
- Что реализовано (суть изменений)
- Статус задачи (Done / Partial / Failed)
- Commit hash
- Push: success / failed
- Deploy: required / not required now
- Ограничения / assumptions
- Следующий рекомендуемый шаг
